# Artifacts

Each run writes one directory under `artifact_dir`:

```text
artifacts/<test_id>/
  manifest.json
  samples.ndjson
  leader-sends.ndjson
  leader-slots-snapshot.json
  leader-slots-snapshot-<start_slot>.json
  summary.json
  summary.md
  report.json
  report.md
  report.html
  comparison.json
  comparison.md
  comparison.html
  index.html
```

## manifest.json

Run-level metadata:

- schema version;
- generated timestamp;
- redacted RPC URL label;
- payer pubkey;
- count/rate settings;
- transaction fee settings;
- provider names and kinds.

Do not put API keys, private keys, or full secret-bearing URLs into the
manifest.

## samples.ndjson

One JSON object per transaction/provider attempt. This is diagnostic sender
metadata, not the primary latency report.

Important fields:

- `test_id`
- `iteration`
- `signature`
- `provider_name`
- `provider_kind`
- `accepted`
- `client_started_at`
- `client_finished_at`
- `client_ack_latency_us`
- `provider_send_started_at`
- `provider_send_finished_at`
- `provider_ack_latency_us`
- `provider_request_id`
- `returned_signature`
- `status_code`
- `error_class`
- `error`
- `route_policy`
- `selected_routes`
- `leader_client_family`
- `leader_software_client`
- `leader_software_client_id`

`provider_ack_latency_us` is useful for diagnosing provider/API behavior, but it
is not landing latency and not first-shred/deshred/processed latency.
`client_ack_latency_us` is the elapsed wall time for the full client submission
round for that transaction and can include concurrent fanout effects.

`leader_client_family` is a coarse cohort label. Route-policy decisions should
prefer `leader_software_client` and `leader_software_client_id` when present.

## leader-sends.ndjson

Generated by `run-leader-paced`.

One JSON object per signed benchmark transaction:

- `test_id`
- `iteration`
- `signature`
- `leader_identity`
- `leader_run_start_slot`
- `leader_run_end_slot`
- `send_slot`
- `sent_at`
- `trigger_source`: `grpc_slot` or legacy `rpc_poll`
- `slot_signal_status`
- `slot_signal_observed_at`
- `leader_client_family`
- `leader_software_client`
- `leader_software_client_id`
- `route_policy`
- `selected_routes`

This file is the public join surface for later leader-cohort enrichment.

## leader-slots-snapshot.json

Generated by `run-leader-paced --capture-leader-slots`.

This file stores the exact JSON-RPC `getLeaderSlots` response captured for the
run in legacy `rpc_poll` mode. In default `grpc_slot` mode, rolling refreshes
are stored as `leader-slots-snapshot-<start_slot>.json`. These files are the
portable cohort-enrichment source for standalone reports.

Top-level fields:

- `schemaVersion`
- `fetchedAt`
- `rpcUrlLabel`, with query strings removed
- `startSlot`
- `limit`
- `response`, the raw JSON-RPC response body

When captured from RPCEdge, `response.result.data[]` can include:

- leader identity and stake;
- data center, city, country, region, and coordinates;
- validator client family/software/version;
- route hints;
- historical landing-latency profiles.

Do not fetch a fresh leader schedule when regenerating a historical report. Use
the saved snapshot so the report reflects what was known during the run.

## Observation Events

The RPCEdge collector and any external observation collectors should emit
newline-delimited `ObservationEvent` JSON.

Fields:

- `schema_version`
- `test_id`
- `signature`
- `source_name`
- `source_kind`: `yellowstone_processed`, `yellowstone_deshred`, or
  `ndjson_import`
- `observed_at`
- `submitted_at`
- `slot`
- `slot_index`
- `source_sequence`

Example:

```json
{"schema_version":1,"test_id":"run-a","signature":"5abc...","source_name":"direct-deshred","source_kind":"yellowstone_deshred","observed_at":"2026-06-29T12:00:00.001Z","submitted_at":"2026-06-29T12:00:00.000Z","slot":429000000,"slot_index":12,"source_sequence":null}
```

## observation-summary.json

Generated by `summarize-observations`.

Source-level outputs:

- observed count;
- missing from matched signatures;
- first-seen count;
- first-seen share;
- submit-to-observed p50/p75/p90/p95/p99/max.

Pair-level outputs:

- matched count;
- win count by source;
- same-timestamp count;
- p50/p75/p90/p95/p99/max delta.

Positive pair delta means `from_source` observed before `to_source`.

## matched-observations.ndjson

Generated by `run-leader-paced --collect-rpcedge`.

This is the RPCEdge observation stream filtered down to signatures generated by
the benchmark run. The runner injects the local benchmark `submitted_at`
timestamp before producing matched summaries.

## matched-observation-summary.json

Generated by `run-leader-paced --collect-rpcedge`.

This is the headline public benchmark summary for the run. It uses only
benchmark-generated signatures, not unrelated stream traffic.

## summary.json

Machine-readable summary grouped by provider:

- count;
- accepted;
- errors;
- min;
- p50;
- p75;
- p90;
- p95;
- p99;
- max.

Percentiles in `summary.json` are based on accepted provider ACK latency and
should be treated as diagnostic only. Percentiles in
`observation-summary.json` are the benchmark result.

## summary.md

Human-readable version of `summary.json`.

## report.json / report.md / report.html

Generated by:

```bash
solana-tx-bench report --artifact-dir artifacts/<test_id>
```

The report is intentionally self-contained. It reads only local files in the
artifact directory:

- `samples.ndjson`
- `leader-sends.ndjson`
- `matched-observations.ndjson`
- `leader-slots-snapshot.json`
- `leader-slots-snapshot-<start_slot>.json`

Leader cohorts are derived from the saved JSON-RPC `getLeaderSlots` responses.
Do not join ClickHouse or fetch a new leader schedule when regenerating a
public report for a historical run.

## comparison.json / comparison.md / comparison.html / index.html

Generated by:

```bash
solana-tx-bench compare \
  --artifact-dir artifacts/<run-a> \
  --label "Run A" \
  --artifact-dir artifacts/<run-b> \
  --label "Run B" \
  --output-dir docs/reports/<comparison-slug>
```

The comparison report is also self-contained. It reads only each run's local
artifact directory and writes a static `index.html` suitable for GitHub Pages.

The scoring model reports five buckets:

- landing latency in milliseconds;
- landing latency in slots;
- processed block position;
- same-slot landing rate;
- observed success ratio.

Provider ACK latency is included only as a diagnostic column and is not part of
landing proof. Compare only runs with the same transaction shape, sender region,
slot trigger, observation source, and fee/tip policy unless the report is
explicitly exploratory.

## Schema Compatibility

The artifact schema is intentionally append-only. Downstream analyzers should
ignore unknown fields so future versions can add:

- confirmation observation IDs;
- provider route details;
- request size;
- response size;
- landed-slot enrichment.
- leader-slot enrichment snapshots;
- gRPC slot trigger details.
