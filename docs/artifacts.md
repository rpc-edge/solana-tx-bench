# Artifacts

Each run writes one directory under `artifact_dir`:

```text
artifacts/<test_id>/
  manifest.json
  samples.ndjson
  summary.json
  summary.md
```

## manifest.json

Run-level metadata:

- schema version;
- generated timestamp;
- redacted RPC URL label;
- payer pubkey;
- count/rate settings;
- transaction fee settings;
- provider names and kinds.

Do not put API keys, private keys, or full secret-bearing URLs into the
manifest.

## samples.ndjson

One JSON object per transaction/provider attempt. This is diagnostic sender
metadata, not the primary latency report.

Important fields:

- `test_id`
- `iteration`
- `signature`
- `provider_name`
- `provider_kind`
- `accepted`
- `client_started_at`
- `client_finished_at`
- `client_ack_latency_us`
- `provider_send_started_at`
- `provider_send_finished_at`
- `provider_ack_latency_us`
- `provider_request_id`
- `returned_signature`
- `status_code`
- `error_class`
- `error`

`provider_ack_latency_us` is useful for diagnosing provider/API behavior, but it
is not landing latency and not first-shred/deshred/processed latency.
`client_ack_latency_us` is the elapsed wall time for the full client submission
round for that transaction and can include concurrent fanout effects.

## leader-sends.ndjson

Generated by `run-leader-paced`.

One JSON object per signed benchmark transaction:

- `test_id`
- `iteration`
- `signature`
- `leader_identity`
- `leader_run_start_slot`
- `leader_run_end_slot`
- `send_slot`
- `sent_at`

This file is the public join surface for later leader-cohort enrichment.

## Observation Events

The RPCEdge collector and any external observation collectors should emit
newline-delimited `ObservationEvent` JSON.

Fields:

- `schema_version`
- `test_id`
- `signature`
- `source_name`
- `source_kind`: `yellowstone_processed`, `yellowstone_deshred`, or
  `ndjson_import`
- `observed_at`
- `submitted_at`
- `slot`
- `slot_index`
- `source_sequence`

Example:

```json
{"schema_version":1,"test_id":"run-a","signature":"5abc...","source_name":"direct-deshred","source_kind":"yellowstone_deshred","observed_at":"2026-06-29T12:00:00.001Z","submitted_at":"2026-06-29T12:00:00.000Z","slot":429000000,"slot_index":12,"source_sequence":null}
```

## observation-summary.json

Generated by `summarize-observations`.

Source-level outputs:

- observed count;
- missing from matched signatures;
- first-seen count;
- first-seen share;
- submit-to-observed p50/p75/p90/p95/p99/max.

Pair-level outputs:

- matched count;
- win count by source;
- same-timestamp count;
- p50/p75/p90/p95/p99/max delta.

Positive pair delta means `from_source` observed before `to_source`.

## matched-observations.ndjson

Generated by `run-leader-paced --collect-rpcedge`.

This is the RPCEdge observation stream filtered down to signatures generated by
the benchmark run. The runner injects the local benchmark `submitted_at`
timestamp before producing matched summaries.

## matched-observation-summary.json

Generated by `run-leader-paced --collect-rpcedge`.

This is the headline public benchmark summary for the run. It uses only
benchmark-generated signatures, not unrelated stream traffic.

## summary.json

Machine-readable summary grouped by provider:

- count;
- accepted;
- errors;
- min;
- p50;
- p75;
- p90;
- p95;
- p99;
- max.

Percentiles in `summary.json` are based on accepted provider ACK latency and
should be treated as diagnostic only. Percentiles in
`observation-summary.json` are the benchmark result.

## summary.md

Human-readable version of `summary.json`.

## Schema Compatibility

The artifact schema is intentionally append-only. Downstream analyzers should
ignore unknown fields so future versions can add:

- confirmation observation IDs;
- provider route details;
- request size;
- response size;
- landed-slot enrichment.
